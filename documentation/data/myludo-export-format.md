# MyLudo Collection Export Format

## Research Status: Spike Report

**Date researched:** 2026-04-16  
**Sources checked:** myludo.fr (public pages), BoardGameGeek forums, Reddit r/boardgames, GitHub issues, community-shared samples  
**Outcome:** The exact field list and delimiter could not be confirmed from a live export because a MyLudo account is required and no definitive format specification is published. This document records what was found and produces a recommended parsing strategy with explicit fallback assumptions.

---

## What Was Found

MyLudo (https://www.myludo.fr) is a French board game collection and tracking platform. Based on community discussion (BGG forum threads, Reddit posts) and the conventions common to French web applications:

- The export feature is accessible from the user's library or profile settings page.
- The download is typically a `.csv` file served with `Content-Type: text/csv` or `application/octet-stream`.
- French software conventionally uses **semicolon (`;`)** as the field delimiter because the comma is used as the decimal separator in French locale, making comma-delimited files ambiguous for numbers.
- The file is expected to be **UTF-8** encoded; a UTF-8 BOM (`\uFEFF`) may be present on files generated by the site for Windows compatibility.

### Most Likely Header Row (Best-Guess)

Based on the French-language UI and community reports, the header row is believed to contain fields similar to:

```
Nom;Type;Note;Commentaire;Tags;Date ajout
```

Where:

| Field | Description |
|-------|-------------|
| `Nom` | Game name (French: "Name") â€” **this is the game title field** |
| `Type` | Category or type label (e.g., "Jeu de base", "Extension") |
| `Note` | User rating (numeric or empty) |
| `Commentaire` | User notes/comment (free text, may contain the delimiter â€” requires quote handling) |
| `Tags` | Comma-separated or free-text tags |
| `Date ajout` | Date the game was added to the collection |

> **Caution:** The exact field names, count, and order are unconfirmed. The parser must treat the header row as authoritative and locate the `Nom` column by name rather than by fixed column index.

### Alternate Field Names Observed

Community members have reported variations, including:
- `name` or `Name` (English-language account setting)
- `titre` (alternative French spelling for "title")
- `jeu` (French: "game")

The parser should accept any of: `Nom`, `nom`, `name`, `Name`, `titre`, `jeu`.

## Encoding

- **Primary assumption:** UTF-8 without BOM
- **Fallback:** UTF-8 with BOM (`\uFEFF` prefix) â€” must be stripped before parsing
- `File.text()` in the browser uses UTF-8 by default; no explicit encoding override is needed
- Latin-1 / Windows-1252 is possible for older exports but is not assumed; if accented characters appear garbled the user should re-export

---

## Delimiter

- **Primary assumption:** Semicolon (`;`)
- **Fallback:** Comma (`,`) for accounts that may use an English locale or for exports from a different MyLudo version

Detection strategy: count semicolons and commas in the first (header) row; use whichever produces more columns (higher count), with semicolon winning ties.

---

## Quoted Fields

Fields that contain the delimiter character or a newline are expected to be surrounded by double-quote characters (`"`), following RFC 4180:

```
"Commentaire contenant; un point-virgule"
```

A double-quote inside a quoted field is escaped as `""`. The parser must handle this.

---

## Edge Cases

| Scenario | Handling |
|----------|----------|
| UTF-8 BOM at start of file | Strip `\uFEFF` before splitting |
| `\r\n` Windows line endings | Normalize to `\n` before splitting |
| Trailing empty line | Filter out lines that are all-whitespace |
| Blank `Nom` value | Skip; do not include empty strings in output |
| `Nom` column contains only whitespace | Trim and skip |
| Quoted `Nom` containing accented characters | Handled by the quote-aware split; accents pass through unchanged |
| File with only a header row and no data rows | Valid; return `{ ok: true, gameNames: [] }` |
| File with no recognisable header | Return `{ ok: false, error: '...' }` |
| Binary / non-text content | Return `{ ok: false, error: '...' }` |
| 0-byte or all-whitespace file | Return `{ ok: false, error: '...' }` |

## Recommended Parsing Strategy

### Parser choice

Use **native `String` methods** (`split`, character-by-character quote scan) rather than a streaming CSV library. The file is fully read into memory via `File.text()` before parsing, so streaming is unnecessary and adding a library dependency is out of scope.

### Algorithm

```
1. Read file: const raw = await file.text()
2. Strip BOM: raw.replace(/^\uFEFF/, '')
3. Reject if empty: if (!content.trim()) â†’ error
4. Normalize line endings: replace \r\n and \r with \n
5. Split into lines: content.split('\n'), filter out blank lines
6. Detect delimiter from first line: count ';' vs ','; prefer ';' on tie
7. Quote-aware split function: walk character-by-character, toggle inQuotes on '"', push field on delimiter when !inQuotes
8. Parse header row: split first line â†’ lowercase-trim each field
9. Find name field index: first match among ['nom', 'name', 'titre', 'jeu']
10. If no name field found â†’ { ok: false, error: 'not a MyLudo file' }
11. Remaining lines are data rows: extract field at nameFieldIndex, trim, skip blank
12. Return { ok: true, gameNames: string[] }
```

### How to detect non-MyLudo files

A file is rejected as unrecognised if, after applying the delimiter detection and header split, no field in the header row matches any of the known name-field identifiers (`nom`, `name`, `titre`, `jeu`). This catches:
- HTML files (`<html>...</html>`)
- JSON files
- Arbitrary text with no valid delimiter structure
- Binary content (will produce a single-field header matching nothing)

### `File.text()` encoding assumption

`File.text()` always decodes as UTF-8 in browsers. No alternative decoding is attempted. The BOM strip handles the most common Windows encoding artefact.
