# Epic 45 — MyLudo Collection Import: Task List

## Story 45.1 — Research and document the MyLudo collection export file format

- [ ] Search the MyLudo website (https://www.myludo.fr) for any documented collection export feature; check user settings, library, profile, and help/FAQ pages for export links, format documentation, or sample output
- [ ] Search community sources (BoardGameGeek forums, Reddit r/boardgames, MyLudo support forum, GitHub issues) for posts or threads describing the MyLudo export format; record any sample file structure, field names, delimiter choices, or encoding details found
- [ ] If a sample export file is obtainable (by creating a test MyLudo account or from a community-shared sample), record the full raw content of the first five lines including the header row verbatim; note the file extension and MIME type offered by the browser download
- [ ] Identify the exact column name or position that contains the board game title in the export output
- [ ] Identify the file encoding (UTF-8, UTF-8 BOM, Latin-1, or other), the delimiter character (comma, semicolon, or tab), whether string fields are quoted, and whether multi-word titles with accented characters are handled correctly
- [ ] Create `documentation/data/myludo-export-format.md` and document: confirmed file extension, MIME type, encoding, delimiter, header row field list, the field containing the board game name, any observed edge cases (quoted fields, trailing commas, blank rows), and the total number of fields per row
- [ ] If the export format cannot be confirmed from any public source, produce a spike report section in `documentation/data/myludo-export-format.md` stating: what sources were checked, what was found, what remains uncertain, and a recommended parsing strategy with explicit fallback assumptions (e.g. semicolon-delimited, game name in column 0 or a named header such as `"title"` or `"name"`)
- [ ] The parsing strategy recommendation in the document must specify: whether to use native `String.split` or a streaming CSV parser, how to handle quoted fields containing the delimiter, how to detect and reject files that are clearly not a MyLudo export (e.g. binary content, XML, unrecognised header row), and which `File.text()` encoding assumption to make
- [ ] Test: review `documentation/data/myludo-export-format.md` manually; confirm it covers all of: file extension, encoding, delimiter, header fields, the game-name field identity, and a concrete parsing strategy recommendation; confirm the document is authored and reviewed before any code changes are made under `src/`
- [ ] QC (Automated): no automated tests apply to this research story; QC should verify `documentation/data/myludo-export-format.md` exists and is non-empty (`wc -c documentation/data/myludo-export-format.md`); run `npm run lint` to confirm no inadvertent source file changes were introduced during research

## Story 45.3 — Parse the uploaded MyLudo export file client-side and extract owned game names

- [ ] Create `src/app/myludo-import-utils.mjs` and export `parseMyludoFile(file)` that accepts a browser `File` object and returns a `Promise<{ ok: boolean, gameNames?: string[], error?: string }>`
- [ ] Read the file content entirely client-side using `await file.text()` (browser File API); do not use `XMLHttpRequest`, `FileReader` callbacks, or any server round-trip; no file data is transmitted anywhere
- [ ] Using the field layout confirmed in `documentation/data/myludo-export-format.md`, detect the delimiter (prefer semicolon, fall back to comma based on whichever produces more consistent column counts in the first row), split into rows, strip the header row by name-matching the expected header fields, and extract the game name from the correct column index or named header field
- [ ] If the file is empty (0-byte content or all-whitespace after trimming), return `{ ok: false, error: 'The selected file is empty. Please export your collection from MyLudo and try again.' }`
- [ ] If the file parses structurally but contains zero data rows (header present, no items), return `{ ok: true, gameNames: [] }` — the caller is responsible for surfacing the empty-collection message
- [ ] If the parsed structure is completely unrecognisable (no delimiter-separated rows matching the expected header layout, or binary content detected), return `{ ok: false, error: 'This file does not appear to be a MyLudo collection export. Please check the file and try again.' }`
- [ ] Strip leading/trailing whitespace from each extracted game name; filter out blank entries; return `{ ok: true, gameNames: string[] }` where each string is a non-empty trimmed game name
- [ ] Wire `parseMyludoFile` into the `actions.importMyludoFile` handler in `src/components/App.svelte`: set `myludoImportStatus` to `'parsing'`, call `await parseMyludoFile(file)`, on failure set status to `'error'`, store the error string, and surface it inline in `CollectionTab` via `myludoImportError` (in addition to any toast notification)
- [ ] Test: in a browser, attempt import of (a) a well-formed MyLudo export file — confirm game names are logged/extracted correctly; (b) an empty `.csv` file — confirm the user-readable error appears inline and no owned sets change; (c) a file with only a header row and no data rows — confirm `gameNames` is empty and no error is shown; (d) a binary or unrecognised file — confirm the graceful error message appears
- [ ] QC (Automated): in `test/epic45-myludo-import.test.mjs`, unit-test `parseMyludoFile` using `new File([content], 'export.csv', { type: 'text/csv' })` constructed inline; cover: (a) a valid CSV string with correct header and two game rows returns `{ ok: true, gameNames: ['Game A', 'Game B'] }`; (b) an empty file (`new File([''], ...)`) returns `{ ok: false }`; (c) a file containing only the header row returns `{ ok: true, gameNames: [] }`; (d) a file with clearly unrecognised content (e.g. `'<html></html>'`) returns `{ ok: false }`

## Story 45.4 — Match extracted game names to the app's expansion catalog using name and alias lookup

- [ ] Export `matchMyludoNamesToSets(myludoGameNames, sets)` from `src/app/myludo-import-utils.mjs`, where `myludoGameNames` is a `string[]` and `sets` is the `runtime.sets` array produced by `createEpic1Bundle`
- [ ] Before iterating over names, build a lookup `Map<string, { setId: string, setName: string }>` keyed on the lowercase-trimmed version of each set's `name` and of each entry in its `aliases` array; build this map once per `matchMyludoNamesToSets` call, not once per name
- [ ] For each `myludoGameName`, normalise to `myludoGameName.trim().toLowerCase()` and look it up in the map; when found, record `{ setId, setName, myludoName: myludoGameName }` in the `matched` array
- [ ] Collect all MyLudo game names that resolve to no catalog entry in an `unmatched` array, preserving their original casing as extracted from the file
- [ ] Deduplicate `matched` by `setId` so that a repeated name in the import file does not produce duplicate catalog entries; keep the first occurrence
- [ ] Return `{ matched: Array<{ setId: string, setName: string, myludoName: string }>, unmatched: string[] }`
- [ ] Wire `matchMyludoNamesToSets` into `actions.importMyludoFile` in `src/components/App.svelte`: call it after `parseMyludoFile` resolves with `ok: true`, passing `bundle.runtime.sets`
- [ ] Test: using the browser dev console with the app loaded, call `matchMyludoNamesToSets(['Legendary: A Marvel Deck Building Game', 'Dark City', 'Unknown Title'], bundle.runtime.sets)` and verify "Core Set" is matched via its alias, "Dark City" is matched by canonical name, and "Unknown Title" appears in `unmatched`; verify passing the same name twice results in only one matched entry
- [ ] QC (Automated): in `test/epic45-myludo-import.test.mjs`, load `src/data/canonical-game-data.json` via `createEpic1Bundle`; assert `matchMyludoNamesToSets(['Legendary: A Marvel Deck Building Game'], sets)` returns `matched[0].setId === 'core-set'` (alias match); assert `matchMyludoNamesToSets(['dark city'], sets)` (all-lowercase) returns `matched[0].setId === 'dark-city'` (case-insensitive canonical match); assert `matchMyludoNamesToSets(['Unknown Expansion'], sets)` yields `matched.length === 0` and `unmatched[0] === 'Unknown Expansion'`; assert passing the same name twice in the input produces exactly one `matched` entry (deduplication)
